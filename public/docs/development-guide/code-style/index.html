代码风格与命名约定

这页不追求给出一套完全外部化的编码规范，而是总结当前仓库中已经能直接观察到的风格线索：如何命名 driver、profile、plugin、runtime 文件，目录边界应该怎样划分，哪些逻辑应放在主仓库，哪些逻辑更适合下沉到插件或 runtime 层，以及哪些基本质量门槛值得长期保持。

1. 先以仓库现有风格为准

这个项目已经给出了一些明确的风格信号。与其额外发明一套规则，不如先尊重仓库现有约定，再在此基础上补充具体建议。

来源 可直接观察到的约定
pyproject.toml 使用 Ruff；`line-length = 100`；`target-version = py311`；lint 选择 `E/F/I/N/W`
hal/base_driver.py Driver 接口采用清晰的动词命名：`load_scene`、`execute_action`、`get_scene`、`get_runtime_state`
hal/drivers/__init__.py 内置 driver 通过短名映射到完整类路径，短名偏向 snake_case
模板与 profile 运行态协议文件使用全大写 Markdown 文件名，如 `ENVIRONMENT.md`、`EMBODIED.md`、`ACTION.md`

2. 命名约定：先保持可预测，再谈个性化

Driver 名称

推荐使用 snake_case 的短名，例如 `go2_edu`、`rekep_real`、`xlerobot_2wheels_remote`。这个名字会直接出现在 Watchdog 的 `--driver` 参数和注册表里。

Python 类名

Driver 类遵循 PascalCase，例如 `Go2Driver`、`RekepRealDriver`、`MyRobotDriver`。模块与类名的区分应尽量稳定、直观。

profile 文件名

建议与 driver 短名保持一致，例如 `go2_edu.md`、`rekep_real.md`。这样在排查 profile 安装问题时，路径关系更容易一眼看懂。

一个很实用的原则是：让使用者在看到 `--driver my_robot`、`my_robot.md`、`my_plugin.driver.MyRobotDriver` 时，能够自然地推断出它们是同一组实现，而不需要额外记忆映射关系。

3. 目录边界：哪些逻辑应放在哪一层

命名问题通常不难，真正容易混乱的是职责边界。目录一旦失去边界，项目很快就会变得难以维护。

位置 更适合放什么 不建议放什么
PhyAgentOS/agent/tools/ 高层能力封装、工具 schema、与 Agent 协作的执行入口 具体硬件 SDK 调用细节
hal/drivers/ 内置 driver 的最小执行适配层 大段第三方 SDK 逻辑和复杂 runtime 流程
hal/profiles/ 能力画像、支持动作、物理约束 运行态数据
插件仓库 runtime/ 真机 bridge、厂商 SDK、预检、dry-run、重依赖和长流程 主仓库级别的通用协议逻辑

4. 代码风格：延续主仓库已有倾向

函数和方法

优先使用语义明确的动词命名，例如 `load_scene`、`execute_action`、`health_check`。方法名应反映对外行为，而不是内部实现细节。

接口层与实现层分离

如果某段逻辑已经明显偏向具体硬件、第三方 SDK 或 runtime 桥接，就不要继续留在 Driver 层。Driver 层越薄，协议边界越清楚。

[tool.ruff]
line-length = 100
target-version = "py311"

[tool.ruff.lint]
select = ["E", "F", "I", "N", "W"]
ignore = ["E501"]

从这些配置可以看出，仓库至少希望保持：

  • 导入顺序清晰（`I`）
  • 命名风格有约束（`N`）
  • 基础语法与常见错误及时暴露（`E/F/W`）

5. 插件仓库的命名与组织建议

如果你在开发外部插件，除了 Python 代码本身，还需要保持仓库级结构的可预测性。

推荐保留的顶层文件

  • PhyAgentOS_plugin.toml
  • README.md / README_zh.md
  • pyproject.toml
  • tests/

推荐的目录边界

  • Python 包目录：存放 `driver.py` 与 `profiles/`
  • runtime/：存放桥接、厂商 SDK 和复杂流程
  • tests/：测试 manifest、driver 动作路由和 runtime 预检

6. 文档与样例也要遵循一致性

代码风格不只是源文件本身，还包括你为新 driver 或插件补充的文档和样例。

内容 建议
命令样例优先给可直接复制执行的完整命令，而不是省略式伪命令
配置样例尽量给最小可用配置，再说明可选字段
动作样例优先给标准 `ACTION.md` fenced JSON，而不是口头描述
profile 样例说明“支持哪些动作”和“有哪些硬约束”，避免只写营销性描述

7. 最低质量门槛：新代码至少应满足什么

不是每次都要把一切做得很完整，但至少应确保新增代码能达到可集成、可调试、可定位问题的基本门槛。

  • Driver 或插件能被成功加载
  • 有对应 profile，且运行时可安装为 `EMBODIED.md`
  • 最少有一组动作或命令验证路径
  • 至少有基础测试或冒烟验证
  • README 能解释安装、启动与调试方式
一个很简单的判断方式是：如果另一个维护者接手你的代码，是否可以只靠仓库内容本身完成安装、理解边界并做一次最小验证。如果不能，通常说明结构和文档还不够自解释。
这一页的目标不是额外发明一套规范，而是把主仓库已经隐含的风格信号整理出来，帮助新增代码保持一致性。
