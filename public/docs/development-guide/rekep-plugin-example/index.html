ReKep 插件开发样例

这一页不只是说明 ReKep 怎么安装，更重要的是把它作为一个完整插件样板来拆开看：清单文件如何声明驱动、Driver 与 runtime 如何分层、本地开发时怎样回到主仓库联调、预检与 dry-run 如何帮助你缩小问题范围，以及为什么这种结构适合作为新插件的起点。

1. 为什么以 ReKep 作为样例

ReKep 不是一个“最简单”的插件，但它非常适合做样板。因为它同时展示了：插件清单、Driver 适配层、runtime 层、部署脚本、真实机器人桥接、测试与文档，这些正是一个成熟外部插件需要具备的核心部分。

边界清楚

主仓库只关心统一 Driver 接口；插件仓库则负责真机 runtime、SDK 和机器人适配细节。

层次完整

从清单文件到 Driver，再到 runtime 命令与本地部署脚本，链路相对完整，适合直接映射到你自己的插件结构。

可直接迁移

如果你准备开发新机器人插件，通常不必从零设计骨架，而是可以直接替换 ReKep 的名字、动作映射和 runtime 实现。

2. 一个插件最小需要哪些文件

ReKep 提供的是“完整样板”，但最小插件不需要一次把所有内容都写满。先把骨架搭好，再逐步补复杂能力，会更稳。

my-phyagentos-plugin/
├── PhyAgentOS_plugin.toml
├── README.md
├── README_zh.md
├── pyproject.toml
├── my_plugin/
│   ├── __init__.py
│   ├── driver.py
│   └── profiles/
│       └── my_robot.md
├── runtime/
│   ├── README.md
│   └── your_runtime_entry.py
└── tests/
    └── test_my_driver.py
路径 职责
PhyAgentOS_plugin.toml声明插件提供哪个 driver、入口类是什么、profile 在哪里
my_plugin/driver.py真正被 Watchdog 加载的 `BaseDriver` 实现
my_plugin/profiles/*.md本体能力档案，启动时会复制为运行时 `EMBODIED.md`
runtime/真机桥接、第三方 SDK、预检和复杂执行逻辑
tests/插件自己的单元测试和冒烟测试

3. 清单文件怎么写

插件能否被发现，首先取决于 `PhyAgentOS_plugin.toml` 是否写得清楚。主仓库真正依赖的是这里给出的 driver 元数据。

[plugin]
name = "my_robot"
version = "0.1.0"
description = "My robot plugin for PhyAgentOS"

[driver]
name = "my_robot"
module = "my_plugin.driver"
class = "MyRobotDriver"
profile_path = "my_plugin/profiles/my_robot.md"

[python]
sys_paths = ["."]

[[requirements]]
path = "runtime/requirements.txt"
optional = false
这里最关键的字段有两个：`driver.name` 决定 Watchdog 启动时用什么名字找到你的插件；`driver.module` + `driver.class` 决定主仓库实际导入哪个 Python 类。

4. ReKep 最值得学习的点：Driver 和 runtime 分层

ReKep 的设计并不是把所有逻辑都塞进 `BaseDriver` 实现里，而是明确分成两层：Driver 层负责适配主仓库协议，runtime 层负责真机执行和复杂流程。

Driver 层负责什么

`ReKepRealDriver` 负责对接 `BaseDriver` 接口：接动作、转参数、返回人类可读结果、暴露 profile 路径。这一层应尽量薄。

runtime 层负责什么

`runtime/dobot_bridge.py` 负责预检、dry-run、真机执行、长程任务与后台行为。这一层更适合承载第三方 SDK 和复杂桥接逻辑。

这种拆法的直接好处是：当你排查问题时，可以更快判断故障是在“协议适配”还是在“机器人真实执行”这一层。

5. 本地开发时怎样把插件接回主仓库

开发插件时，最常见的结构是把插件仓库放在主仓库同级目录，然后通过本地路径安装回主仓库。

git clone https://github.com/SYSU-HCP-EAI/PhyAgentOS.git
git clone https://github.com/baiyu858/PhyAgentOS-rekep-real-plugin.git

cd PhyAgentOS
pip install -e .

python scripts/deploy_rekep_real_plugin.py \
  --repo-url ../PhyAgentOS-rekep-real-plugin

这类联调方式的好处是：你可以修改插件仓库代码，同时依旧使用主仓库的 Watchdog、Agent 和工作区机制做全链路验证。

6. 开发后如何验证：先 preflight，再 dry-run，再全链路

ReKep 的验证路径非常适合作为插件开发的标准流程：先缩小问题范围，再逐步扩大到完整系统。

# 预检
python runtime/dobot_bridge.py preflight --pretty

# dry-run
python runtime/dobot_bridge.py execute \
  --instruction "pick up the red block and place it on the tray" \
  --pretty

# 真机执行
python runtime/dobot_bridge.py execute \
  --instruction "pick up the red block and place it on the tray" \
  --execute_motion \
  --pretty
# 最后再回到完整链路
paos onboard
python hal/hal_watchdog.py --driver rekep_real --workspace ~/.PhyAgentOS/workspace
paos agent
一个稳妥的原则是：不要一开始就把问题抛给全链路。先确认 runtime 自己成立，再确认 Driver 能路由动作，最后再回到 Agent 和工作区协议层验证整体闭环。

7. ReKep 如何同时兼容专用动作和通用动作

ReKep 之所以适合做样板，还因为它同时支持插件原生动作协议和主仓库通用动作协议，这对新插件很有参考价值。

插件原生动作

如 `real_preflight`、`real_execute`、`real_execute_background`、`real_scene_qa`、`real_longrun_start` 等，直接映射到 runtime 子命令。

通用高层动作

如 `move_to`、`pick_up`、`put_down`、`push`、`point_to`、`open_gripper`、`close_gripper`，它们会被转换成自然语言 instruction 后再落到统一执行入口。

如果你正在设计新插件，通常也建议保留这两层：一层给主仓库的通用动作协议，一层给插件内部的专用能力接口。

8. 一个更稳妥的社区开发流程

顺序 建议动作 原因
1先写 `PhyAgentOS_plugin.toml`让插件最先具备可发现性
2再写最小 `BaseDriver`先让 Watchdog 能加载起来
3补 `profile`让运行时 `EMBODIED.md` 能正确安装
4把复杂能力放进 `runtime/`避免把所有逻辑塞进 driver
5先跑 `preflight` 和 `dry-run`优先缩小故障范围
6最后接入 Watchdog 和 Agent再做全链路联调
这一页把 ReKep 当作“结构样板”而不是单纯的部署案例来看，目的是帮助你更快搭起自己的外部插件骨架。
