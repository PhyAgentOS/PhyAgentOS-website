新 Driver 接入流程

这一页聚焦“如何把一个新的本体接入 HAL”。它的目标不是讲概念，而是给出一条足够明确的落地路径：从 `BaseDriver` 的最小接口开始，补 profile、注册 driver、验证 Watchdog 能否加载，再到如何做全链路联调和最小测试。

1. 先理解 Driver 契约

所有新 Driver 的起点都是统一的 `BaseDriver` 接口。只要你的实现遵守这个契约，Watchdog 就可以像使用内置 driver 一样加载它。

class BaseDriver(ABC):
    def get_profile_path(self) -> Path: ...
    def load_scene(self, scene: dict[str, dict]) -> None: ...
    def execute_action(self, action_type: str, params: dict) -> str: ...
    def get_scene(self) -> dict[str, dict]: ...

    def connect(self) -> bool: ...
    def disconnect(self) -> None: ...
    def is_connected(self) -> bool: ...
    def health_check(self) -> bool: ...
    def get_runtime_state(self) -> dict[str, Any]: ...
方法 是否必须 职责
get_profile_path()必须告诉系统当前 driver 使用哪份 `EMBODIED.md` profile
load_scene()必须用 `ENVIRONMENT.md` 解析出的 scene 初始化内部状态
execute_action()必须执行原子动作并返回人类可读字符串
get_scene()必须返回当前环境状态，用于回写 `ENVIRONMENT.md`
get_runtime_state()可选但常用暴露连接状态、导航状态、地图或运行时扩展信息

2. 先写最小可加载 Driver

接入新本体时，不要一开始就把所有能力写完。更稳妥的做法是先让 Watchdog 能够成功加载这台 driver，再逐步补动作和运行态。

from pathlib import Path
from hal.base_driver import BaseDriver

class MyRobotDriver(BaseDriver):
    def __init__(self, gui: bool = False, **kwargs):
        self._scene = {}

    def get_profile_path(self) -> Path:
        return Path(__file__).resolve().parent / "profiles" / "my_robot.md"

    def load_scene(self, scene: dict[str, dict]) -> None:
        self._scene = dict(scene)

    def execute_action(self, action_type: str, params: dict) -> str:
        return f"executed {action_type}"

    def get_scene(self) -> dict[str, dict]:
        return dict(self._scene)
这个最小实现的目标不是“能做很多事”，而是先确认主仓库能够稳定导入 driver、安装 profile，并顺利完成一轮动作消费与状态回写。

3. 补上 profile，而不是把能力说明塞进代码里

Driver 只是执行适配层，机器人的能力说明应当放进 profile。这样 Critic 才能在动作派发前判断这台机器人的能力边界。

profile 里通常要写什么

  • Identity：机器人名称、类型、driver/profile
  • Sensors 或 Degrees of Freedom
  • Supported Actions
  • Physical Constraints
  • Connection 与 Runtime Protocol 说明

为什么不能缺

没有 profile，主仓库仍然可能导入 driver，但 Critic 会失去动作约束依据，`ROBOTS.md` 也无法给出可靠能力摘要。

4. 内置 driver 如何注册

如果你要做的是主仓库内置 driver，而不是外部插件，那么除了创建 driver 文件，还需要在内置注册表里登记短名。

DRIVER_REGISTRY: dict[str, str] = {
    "simulation": "hal.drivers.simulation_driver.SimulationDriver",
    "go2_edu": "hal.drivers.go2_driver.Go2Driver",
    "xlerobot_2wheels_remote": "hal.drivers.xlerobot_2wheels_remote_driver.XLerobot2WheelsRemoteDriver",
    "my_robot": "hal.drivers.my_robot_driver.MyRobotDriver",
}

如果是外部插件，则由插件清单和本地 registry 负责发现，不需要修改主仓库内置的 `DRIVER_REGISTRY`。

5. Watchdog 实际上会怎样使用你的 Driver

理解 `hal_watchdog.py` 的启动顺序，有助于你判断问题到底出在导入、profile 安装、动作消费还是环境回写阶段。

顺序 Watchdog 行为 对 Driver 的要求
1加载 driverdriver 可以被正确 import 并实例化
2安装 profile 到运行时 `EMBODIED.md``get_profile_path()` 返回有效路径
3尝试建立连接并刷新健康状态`connect()` / `health_check()` 行为合理
4读取 `ACTION.md` 并调用 `execute_action()`动作类型和参数能够被接住
5调用 `get_scene()` / `get_runtime_state()` 回写运行态可被序列化进 `ENVIRONMENT.md`

6. 新 Driver 最实用的调试顺序

1. 先让 driver 能被 import 和实例化
2. 再确认 profile 可以被安装到 EMBODIED.md
3. 再确认 ACTION.md 中的动作能进入 execute_action()
4. 再确认 get_scene() / get_runtime_state() 能回写 ENVIRONMENT.md
5. 最后再用 paos agent 做完整自然语言链路联调
调试顺序的核心思路是：先确认“能加载”，再确认“能执行”，最后确认“能回写”。如果一开始就上完整链路，问题来源会变得很难切分。

7. 最后一层验证：回到完整闭环

最小 Driver 跑通之后，最终仍然要回到完整闭环去验证它是否真的能和 Agent、Critic 与工作区协议协同。

paos onboard
python hal/hal_watchdog.py --driver my_robot --workspace ~/.PhyAgentOS/workspace
paos agent

然后用一条最简单、最明确的自然语言任务去触发动作，例如“向前移动一点”或“连接机器人并汇报当前状态”。观察重点是：

  • `ACTION.md` 是否按预期生成和更新
  • `ENVIRONMENT.md` 是否出现你这台机器人的运行态
  • `LESSONS.md` 是否记录了动作拒绝原因（如果被拒绝）
这一页更适合作为新本体接入 HAL 时的实操清单使用：先满足最小契约，再补能力，再做全链路联调。
