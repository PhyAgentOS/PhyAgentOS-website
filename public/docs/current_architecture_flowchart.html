1. 用户与通道输入

来自 CLI 或 gateway 通道的请求首先被标准化成统一消息事件。

  • `paos agent` / `paos gateway`
  • Telegram / Slack / QQ / WhatsApp
  • `InboundMessage` -> `MessageBus`

2. AgentLoop 组装上下文

系统读取会话历史、记忆和 workspace 中的协议文件，构成本轮推理输入。

  • `ContextBuilder`
  • `SessionManager`
  • `MEMORY.md` / `HISTORY.md`
  • `ENVIRONMENT.md` / `EMBODIED.md` / `ROBOTS.md`

3. 模型选择工具链

LLM 不直接调用硬件接口，而是通过工具把高层目标表达成可验证的操作意图。

  • `query_scene_graph`
  • `semantic_navigate`
  • `target_navigation`
  • `execute_robot_action`

4. 语义任务映射为协议动作

系统将目标、位姿和机器人路由信息组织成标准动作参数，并准备写入动作队列。

  • 解析 target / zone / goal_pose
  • 解析 robot workspace
  • 落到 `ACTION.md` 的标准 schema

5. Critic 决定动作是否放行

动作真正落盘前，需要对照能力约束与当前环境进行安全与可执行性校验。

  • 读取 `EMBODIED.md`
  • 读取 `ENVIRONMENT.md`
  • `VALID` 才能进入队列
  • `INVALID` 写入 `LESSONS.md`

6. HAL Watchdog 与 Driver 执行

执行侧轮询动作队列，通过驱动适配到仿真器或真实机器人，并更新运行状态。

  • `hal_watchdog.py`
  • `BaseDriver` + built-in / plugin driver
  • 执行导航、移动、操作等原子动作

文件协议层是系统边界

PhyAgentOS 当前最关键的结构设计，不是某个类，而是 workspace 文件协议。

  • `EMBODIED.md`：能力与约束
  • `ENVIRONMENT.md`：环境、scene graph、robot runtime state
  • `ACTION.md`：待执行动作队列
  • `LESSONS.md`：失败经验与拒绝记录
  • `ROBOTS.md`：fleet 模式下的机器人索引

回环逻辑

该系统不是单向命令下发，而是标准的“状态闭环”。

  • 执行前：Agent 读取环境与能力
  • 执行中：Watchdog 消费动作并调用 driver
  • 执行后：动作结果和环境变化回写文件
  • 下一轮：Agent 再读取新状态继续推理

具身扩展方式

设备差异被 `BaseDriver` 接口与插件机制吸收，因此上层认知链可以跨本体复用。

  • 内置 driver：simulation / go2 / xlerobot
  • 外部 plugin driver：manifest + registry 动态接入
  • fleet 模式：由 `EmbodimentRegistry` 做路径路由

示例主链：让 `go2_edu_001` 走到最近的苹果附近

在这个真实场景里，系统通常会先读取 scene graph 找到苹果，再通过 `semantic_navigate` 生成目标位姿，随后交给 `execute_robot_action` 做 Critic 校验，最终写入目标机器人的 `ACTION.md`，由 watchdog 取出并执行，最后把结果更新回 `ENVIRONMENT.md`。

用户请求 → 读取 `ENVIRONMENT.md` / `ROBOTS.md` → `query_scene_graph` 定位苹果 → `semantic_navigate` 生成 `goal_pose` → Critic 校验 → 写入 `ACTION.md` → Watchdog + Driver 执行 → 回写 `ENVIRONMENT.md`
