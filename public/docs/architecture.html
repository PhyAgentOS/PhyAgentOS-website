PhyAgentOS 框架设计

想象你对着一台机器人说"把桌上的苹果放到篮子里"。这句话不会直接变成电机指令——中间要经过理解、规划、校验、格式化、执行、回传状态六个环节。PhyAgentOS 就是要把这六个环节串成一条清晰可观测的流水线，而不是黑盒。

一句话看懂：数据怎么流

别急着看源码目录。先把这条主线印在脑子里：用户的一句话 → Agent 的规划 → 写进文件 → Watchdog 读取 → 驱动执行 → 状态再写回文件。整个循环围绕着几个 Markdown 文件转。

六个步骤：从一句话到机器人动作

下面按真实执行顺序拆开。如果你调试时卡住了，就对照这六步定位：问题出在理解层？校验层？还是执行层没回写状态？

1. 用户说话 → 2. Agent 想上下文 → 3. 决定要做什么动作 → 4. Critic 拦一下 → 5. 写成 ACTION.md → 6. Watchdog 执行

步骤 1~3：认知层

用户通过 CLI 或网关发任务，AgentLoop 把会话历史、工作区状态拼成提示词，让 LLM 决定调用哪个工具。涉及物理世界时，通常命中 execute_robot_action 或 target_navigation。

步骤 4：Critic 校验

Critic 会扒开 EMBODIED.md 和 ENVIRONMENT.md，看看这个动作是否合法、可达。不合法就直接拒掉，把原因写进 LESSONS.md。

步骤 5：落盘

校验通过后，动作不会直接 RPC 到硬件，而是被格式化成 JSON，写进 ACTION.md。这个动作队列就是认知层留给执行层的唯一接口。

步骤 6：执行与回写

HAL Watchdog 轮询 ACTION.md，发现有 pending 动作就交给 Driver。执行完后，Driver 把最新环境状态回写到 ENVIRONMENT.md，完成闭环。

文件不是日志，是正式接口

很多系统用文件做日志，PhyAgentOS 用文件做协议。不同进程之间不直接调函数，而是读同一份 Markdown。这让整个系统变得像玻璃房一样——你随时能打开文件看当前状态。

三层结构

把代码目录硬背下来没用，建立分层视角更重要。看到一段代码时，先问自己：这是在替机器人"动脑"，还是在"跑腿"？

认知层

PhyAgentOS/agent、providers、cli

负责理解任务、规划动作、调用工具、做安全判断。不碰具体机器人 SDK。

协议层

workspace / workspaces

用 Markdown 承载共享状态，让 Agent、Watchdog、Critic 都能读同一份上下文。

执行层

hal/、drivers、navigation、perception

负责连接硬件、执行动作、刷新环境。不做高层规划。

Single 与 Fleet：同一套代码，两种跑法

Single

一台机器人，一个工作区。调试最爽的方式，本地跑仿真器也是这个模式。

Fleet

多台机器人共享一个 ENVIRONMENT.md，但各自有独立的 ACTION.md 和 EMBODIED.md。

为什么要用 Markdown 做协议？

因为人要看，模型也要看。一个 ENVIRONMENT.md 打开就能读，LLM 也能直接塞进提示词。如果换成二进制 protobuf，调试时你会想哭。

  • 人类可读：出问题先 cat 文件，不用attach debugger。
  • 模型可读：Markdown 天然适合放进 LLM 上下文。
  • 进程间松耦合：Agent 和 Watchdog 不需要 RPC，只要约定文件格式。
记住这条主线：模型只写文件，Watchdog 只读文件执行。中间隔着 Markdown，这就是 PhyAgentOS 最核心的设计。
