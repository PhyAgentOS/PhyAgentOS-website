1. User & Channel Input

Requests from CLI or gateway channels are first normalized into unified message events.

  • `paos agent` / `paos gateway`
  • Telegram / Slack / QQ / WhatsApp
  • `InboundMessage` → `MessageBus`

2. AgentLoop Assembles Context

The system reads conversation history, memory, and workspace protocol files to construct the current reasoning input.

  • `ContextBuilder`
  • `SessionManager`
  • `MEMORY.md` / `HISTORY.md`
  • `ENVIRONMENT.md` / `EMBODIED.md` / `ROBOTS.md`

3. Model Selects Tools

The LLM does not call hardware interfaces directly; instead, it expresses high-level goals as verifiable operation intents through tools.

  • `query_scene_graph`
  • `semantic_navigate`
  • `target_navigation`
  • `execute_robot_action`

4. Semantic Task → Protocol Action

The system organizes goals, poses, and robot routing information into standard action parameters and prepares to write them to the action queue.

  • Parse target / zone / goal_pose
  • Parse robot workspace
  • Map to `ACTION.md` standard schema

5. Critic Gate Validation

Before an action is committed, it is checked against capability constraints and the current environment for safety and feasibility.

  • Read `EMBODIED.md`
  • Read `ENVIRONMENT.md`
  • `VALID` to enter the queue
  • `INVALID` writes to `LESSONS.md`

6. HAL Watchdog & Driver Execution

The execution side polls the action queue, adapts through drivers to simulators or real robots, and updates runtime status.

  • `hal_watchdog.py`
  • `BaseDriver` + built-in / plugin driver
  • Execute navigation, movement, manipulation primitives

File Protocol Layer as System Boundary

The most critical structural design in PhyAgentOS is not a specific class, but the workspace file protocol.

  • `EMBODIED.md`: capabilities & constraints
  • `ENVIRONMENT.md`: environment, scene graph, robot runtime state
  • `ACTION.md`: pending action queue
  • `LESSONS.md`: failure experience & rejection records
  • `ROBOTS.md`: robot fleet index

Feedback Loop Logic

This system is not a one-way command pipeline, but a standard "state closed loop".

  • Pre-execution: Agent reads environment & capabilities
  • During execution: Watchdog consumes actions & invokes drivers
  • Post-execution: Action results & environment changes are written back to files
  • Next iteration: Agent reads the updated state and continues reasoning

Embodiment Extensibility

Device differences are absorbed by the `BaseDriver` interface and plugin mechanism, allowing the upper cognitive chain to be reused across different embodiments.

  • Built-in drivers: simulation / go2 / xlerobot
  • External plugin drivers: manifest + registry for dynamic integration
  • Fleet mode: `EmbodimentRegistry` handles route dispatching

Example: Navigate `go2_edu_001` to the Nearest Apple

In this real-world scenario, the system typically reads the scene graph to locate the apple, generates a target pose via `semantic_navigate`, passes it to `execute_robot_action` for Critic validation, writes to the target robot's `ACTION.md`, which the watchdog picks up and executes, and finally updates the result back to `ENVIRONMENT.md`.

User request → Read `ENVIRONMENT.md` / `ROBOTS.md` → `query_scene_graph` locates apple → `semantic_navigate` generates `goal_pose` → Critic validation → Write `ACTION.md` → Watchdog + Driver execution → Write back `ENVIRONMENT.md`
