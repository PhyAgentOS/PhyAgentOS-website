PhyAgentOS 框架设计

这一页尝试回答一个更基础的问题：PhyAgentOS 作为一套具身框架，究竟是如何组织起来的。它会依次说明系统为何要把认知、协议和执行拆成不同层级，工作区文件为什么不是附属产物，以及一条任务请求如何从推理流转为可执行动作，再回到下一轮上下文中。

认知编排层
工作区协议层
安全校验层
HAL 执行层
反馈闭环

如果只看目录结构，这套系统容易显得分散；如果先按运行顺序来看，理解会简单得多。下面这六个步骤对应一次典型任务从进入系统到完成回写的全过程，也是阅读源码和排查行为时最值得先建立的主线。

1. 用户任务进入系统

入口通常来自 CLI、网关或外部消息渠道。用户表达的是任务目标，而不是底层控制语句。

  • 自然语言请求
  • 单轮调用或多轮会话
  • 入口由 paos agent / paos gateway 承载

2. Agent 构建认知上下文

AgentLoop 读取会话、记忆、历史与工作区状态，形成这一轮真正参与推理的输入。

  • 历史消息与 memory consolidation
  • 工具注册与 tool schema
  • single / fleet 的工作区解析

3. 形成动作意图

如果任务涉及物理世界，模型不会直接碰硬件，而是选择具身动作工具或更高层导航工具。

  • execute_robot_action
  • target_navigation
  • semantic_navigation

4. Critic 做校验

动作进入执行层前，必须结合机器人能力描述与环境状态判断是否合法、可达、可执行。

  • 读取 EMBODIED.md
  • 读取 ENVIRONMENT.md
  • 拒绝危险或不支持动作

5. 文件协议派发动作

通过校验后，动作不会直接 RPC 到硬件，而是被规范化写入 ACTION.md。

  • 动作显式落盘
  • 队列结构可调试、可审计
  • single / fleet 路径由 registry 决定

6. Watchdog 与 Driver 执行

HAL Watchdog 轮询动作文档，交给 Driver 执行，并将动作状态与环境状态回写回系统。

  • 调用 driver.execute_action()
  • 更新 ACTION.md 状态
  • 更新 ENVIRONMENT.md 运行态

工作区协议层是系统边界

这个框架最重要的设计并不是某一个类，而是围绕工作区文件建立起来的共享状态面。模型、开发者和守护进程都通过它理解当前系统状态。

  • ENVIRONMENT.md：环境真相源
  • EMBODIED.md：能力与约束
  • ACTION.md：动作队列
  • LESSONS.md：失败经验

为什么不让 LLM 直控硬件

如果模型直接进入硬件调用链，系统会更难审计，也更难做能力约束和多机器人路由。先标准化、再校验、再落盘，是为了保留清晰的执行边界。

HAL 层只处理执行事实

执行层不承担任务理解，它只处理设备连接、动作执行和状态回写。职责越单纯，迁移到不同机器人时的成本就越可控。

1. 设计目标：把具身系统从黑盒控制改造成可解释流程

PhyAgentOS 关心的并不是“把机器人控制得更像一个黑盒产品”，而是建立一套结构清楚、状态可见、适合持续扩展的具身运行框架。这里的重点在于分工是否明确，以及系统边界是否稳定。

目标一：透明

关键运行态优先放进工作区文件，而不是完全藏在进程内存和对象图里。这样系统更容易调试、解释和演示。

目标二：解耦

认知层不应该知道每台机器人 SDK 的全部细节；执行层也不应该承担任务规划。两者通过协议连接，而不是互相侵入。

目标三：迁移

同一套高层工作流应当可以跑在仿真器、四足平台、机械臂或插件驱动之上，而不是每换一种本体都重写整套系统。

一句话概括：这套框架更强调结构清晰和边界可控，而不是把所有能力都压进一条直接控制硬件的调用链里。

2. 三层结构：认知层、协议层、执行层

理解这套系统时，与其先记目录名称，不如先建立分层视角。这样在阅读代码或排查行为时，更容易知道某个模块属于哪一类职责。

层级 代表模块 主要职责 明确不负责什么
认知层（Track A） PhyAgentOS/cli、PhyAgentOS/agent、PhyAgentOS/providers 理解任务、规划动作、调用工具、做逻辑约束与安全判断 不直接关心具体机器人 SDK 调用细节
协议层 workspace / workspaces 中的 Markdown 与 fenced JSON 文件 承载共享状态、建立多进程和多机器人的一致接口 不负责复杂业务逻辑本身
执行层（Track B） hal/hal_watchdog.py、hal/drivers、hal/navigation、hal/perception 执行动作、维护连接、回写环境与运行态 不负责高层自然语言规划

3. 工作区协议：文件不是附属物，而是正式接口

如果只看代码而不看工作区文件，通常会对系统行为形成偏差理解。这里的文件协议不是附加说明，而是跨层协作真正依赖的接口面。

文件 作用 谁主要写入 谁主要读取
ENVIRONMENT.md 保存环境对象、场景图、机器人运行态、地图或导航状态 Watchdog / 感知与导航模块 Agent / Critic / 查询工具
EMBODIED.md 描述当前机器人实例的能力边界与约束 Watchdog 启动时从 profile 安装 Critic / 具身动作工具
ACTION.md 动作队列文档，是认知层发给执行层的正式请求 EmbodiedActionTool HAL Watchdog
LESSONS.md 记录失败尝试与拒绝原因，形成经验反馈 Critic 拒绝逻辑或失败记录逻辑 Agent / 开发者
ROBOTS.md 多机器人实例、workspace、连接状态与能力摘要索引 EmbodimentRegistry Planner / 运维 / 开发者

模板、Profile、运行态文件的关系

模板定义结构，profile 定义某类机器人能力，运行时文件则承载当前实例真实状态。这三者职责不同，但在执行链路中会连续起作用。

为什么坚持 Markdown-first

因为它天然兼顾人类可读性、模型可读取性和进程间轻量协作能力。对可调试具身系统来说，这种可读性价值很高。

4. 运行时主链路：任务如何变成物理动作

下面按真实执行顺序把一次动作生命周期拆开。理解这条链路，是理解整个框架的关键。

典型任务过程

例如用户说“看看桌子上有什么，然后靠近苹果”。系统不会直接把这句自然语言发给机器人，而是经历规划、工具调用、能力校验、动作落盘、Watchdog 执行和环境回写等阶段。

用户输入任务 → AgentLoop 构建上下文 → 选择具身工具 → Critic 校验 → 写入 ACTION.md → Watchdog 消费执行 → 回写 ENVIRONMENT / LESSONS 
# 典型启动顺序
paos onboard
python hal/hal_watchdog.py --driver simulation
paos agent

# 然后在 agent CLI 中输入任务
# 系统进入：规划 -> 校验 -> 落盘 -> 执行 -> 回写 的闭环
关键点：系统真正的循环不是“模型输出一次就结束”，而是“执行结果持续回流到认知层，成为下一轮推理输入”。

5. 核心模块分工：从目录映射到运行行为

开发者理解框架最快的方法之一，就是把目录结构与真实运行职责对应起来。

模块 作用 阅读时应重点关注
PhyAgentOS/cli/commands.py 系统命令入口，决定 onboard、agent、gateway 怎样启动 配置加载、workspace 初始化、AgentLoop 构造
PhyAgentOS/agent/loop.py 主消息处理引擎，负责上下文构建、工具注册和 LLM 工具迭代 工具是如何进入主流程的
PhyAgentOS/agent/tools/embodied.py 具身动作统一入口，负责路径解析、Critic 校验与动作派发 为什么动作先校验再写入 ACTION.md
PhyAgentOS/embodiment_registry.py 管理 single / fleet 的工作区路径与实例索引 共享工作区和机器人工作区如何映射
hal/hal_watchdog.py 执行层主循环，负责加载 driver、消费动作和回写环境 动作消费、profile 安装、环境刷新
hal/base_driver.py 定义 Driver 的最小能力边界 接入新硬件时最少要实现哪些方法

6. Single 与 Fleet：同一架构下的两种运行拓扑

PhyAgentOS 不只支持单机器人本地验证，也支持多机器人共享环境。理解 single 与 fleet 的路径差异，对调试和扩展都很重要。

Single 模式

一个工作区即可跑通全链路。Agent 与 Watchdog 围绕同一 workspace 协作，适合本地调试、仿真和快速验证。

Fleet 模式

一个 shared workspace 保存全局环境和任务状态，多台机器人各自持有独立的 ACTION.md 与 EMBODIED.md。

对比项 Single Fleet
环境状态 当前工作区 ENVIRONMENT.md shared workspace ENVIRONMENT.md
动作文件 当前工作区 ACTION.md 目标机器人工作区 ACTION.md
能力文件 当前工作区 EMBODIED.md 目标机器人工作区 EMBODIED.md
适用场景 本地开发、仿真、单机 demo 多机器人协作、共享环境任务

7. 扩展边界：为什么新硬件和新能力容易接入

如果一个具身框架每接一种本体都要修改 Agent 主链路，它很快会变得不可维护。PhyAgentOS 通过最小 Driver 抽象和外部插件机制降低了扩展成本。

接新 Driver

只要实现最小执行接口，并提供 profile，就能把新机器人接入统一执行协议中。

接新 Tool

高层工具可以继续复用具身动作分发，而不需要直接碰硬件控制细节。

接外部插件

通过插件清单注册外部驱动，使设备适配仓库能够独立演进，而不强绑核心仓库版本。

class BaseDriver(ABC):
    def get_profile_path(self) -> Path: ...
    def load_scene(self, scene: dict[str, dict]) -> None: ...
    def execute_action(self, action_type: str, params: dict) -> str: ...
    def get_scene(self) -> dict[str, dict]: ...
扩展价值：认知层只需要知道“有一个遵守统一动作契约的执行体”，而不需要理解每台机器人底层 SDK 的特殊写法。

8. 设计权衡：强调可解释性，而不是极致实时性

这套架构明显偏向可解释、可审计、可迁移和可扩展，因此它更像上层认知与任务编排框架，而不是毫秒级实时控制器。

维度 收益 代价
透明性 运行态可直接观察，适合调试、演示和研究 文件协议比纯内存调用更冗长
解耦性 硬件接入成本更低，支持多本体与插件扩展 需要更严格维护协议一致性
安全性 动作可在进入执行层前被校验 多一层约束意味着链路更长
可迁移性 仿真、真机和插件共享同一高层工作流 底层动作必须先被抽象为统一语义
适用边界：如果你的目标是“高层任务编排 + 可解释具身闭环”，这套架构非常合适；如果你追求硬实时控制，应把实时控制下沉到底层控制器，让 PhyAgentOS 保持在更上层的认知和编排位置。
这一页的重点不是把结构名词一次性讲全，而是帮助读者先建立一条稳定的理解路径：先看系统如何分层，再看状态如何流动，最后再回到具体模块与扩展边界。
