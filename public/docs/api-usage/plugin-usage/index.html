插件使用说明

本页聚焦插件在运行时的使用方式，而不是插件如何开发。重点包括：PhyAgentOS 如何发现插件、部署脚本实际做了什么、ReKep 插件怎样接入 HAL Watchdog、如何直接通过 `ACTION.md` 或 runtime 级命令调试插件，以及这些调用方式分别适合什么阶段使用。

1. 插件如何被系统发现

在运行时，外部插件并不是“自动就能用”。主仓库需要先知道插件提供了哪个 driver、入口类在哪里、profile 在哪里，以及加载时要把哪些路径加入 Python 搜索路径。

步骤 发生位置 作用
1 PhyAgentOS_plugin.toml 插件仓库通过清单声明 `driver.name`、`driver.module`、`driver.class` 和 `profile_path`
2 scripts/deploy_rekep_real_plugin.py 把插件 clone 或复制到本地插件目录，并安装依赖
3 hal/plugins.py 读取清单并把解析后的驱动信息写入本地 registry
4 hal/drivers/__init__.py 在内置驱动表中找不到同名 driver 时，从外部 registry 动态导入
5 hal/hal_watchdog.py --driver 像使用内置 driver 一样启动该插件 driver
插件可以理解为“独立仓库中的 HAL driver + profile + runtime”。主仓库只关心统一接口，并不关心你插件内部的第三方 SDK 细节。

2. ReKep 插件的安装方式

ReKep 插件是目前最适合参考的外部插件样例。无论是线上安装还是本地联调，它的安装方式都很清晰。

在线安装

cd PhyAgentOS
python scripts/deploy_rekep_real_plugin.py \
  --repo-url https://github.com/baiyu858/PhyAgentOS-rekep-real-plugin.git

本地联调安装

cd PhyAgentOS
python scripts/deploy_rekep_real_plugin.py \
  --repo-url ../PhyAgentOS-rekep-real-plugin
# 带 solver 依赖的安装
python scripts/deploy_rekep_real_plugin.py \
  --repo-url https://github.com/baiyu858/PhyAgentOS-rekep-real-plugin.git \
  --with-solver

# 只注册不安装依赖
python scripts/deploy_rekep_real_plugin.py \
  --repo-url ../PhyAgentOS-rekep-real-plugin \
  --no-install-deps
如果你的目标是“先把插件跑起来”，在线安装通常最直接；如果你的目标是改插件代码并反复联调，本地路径安装会更高效。

3. 通过 HAL Watchdog 调用插件

这是最接近完整运行链路的使用方式。Agent 负责生成动作，Watchdog 负责监听 `ACTION.md` 并调用插件 driver 执行。

paos onboard

# 终端 1
python hal/hal_watchdog.py --driver rekep_real --workspace ~/.PhyAgentOS/workspace

# 终端 2
paos agent

在这个模式下，Agent 生成的动作会先写入 `ACTION.md`，再由 `rekep_real` 驱动像内置 driver 一样处理。这是验证“感知—决策—执行”完整闭环时最推荐的方式。

4. 通过 ACTION.md 直接调用插件

如果你不想先经过 Agent 推理，而是想直接验证某个动作是否能被插件接住，可以手动写入 `ACTION.md`。这适合排查动作映射问题。

插件原生动作

{
  "action_type": "real_execute",
  "parameters": {
    "instruction": "pick up the chili pepper and place it in the plate",
    "execute_motion": true
  },
  "status": "pending"
}

通用高层动作

{
  "action_type": "pick_up",
  "parameters": {
    "target": "red_apple"
  },
  "status": "pending"
}

前者使用插件自己的专用动作协议，后者使用主仓库通用动作协议。ReKep 同时兼容这两类方式，因此非常适合作为使用样例。

5. 直接调 runtime 进行调试

插件开发阶段，最常见的需求是“先绕开 Agent 和 Markdown 协议，单独验证 runtime 本身”。这时直接调用 runtime 会更高效。

# 预检
python runtime/dobot_bridge.py preflight --pretty

# dry-run
python runtime/dobot_bridge.py execute \
  --instruction "pick up the red block and place it on the tray" \
  --pretty

# 真机执行
python runtime/dobot_bridge.py execute \
  --instruction "pick up the red block and place it on the tray" \
  --execute_motion \
  --pretty
# 切换 robot family 的示例
python runtime/dobot_bridge.py preflight \
  --robot_family cellbot \
  --robot_driver your_driver
runtime 级调试最适合定位第三方 SDK、桥接脚本或机器人适配器的问题，因为它绕开了 Agent 和工作区协议，可以更快缩小故障范围。

6. ReKep 插件内部结构，为什么值得参考

ReKep 的价值不只在于“它能用”，更在于它展示了一个比较成熟的外部插件应该怎样分层。

Driver 层

`phyagentos_rekep_real_plugin/driver.py` 负责对接 PhyAgentOS 的 `BaseDriver` 接口，是主仓库能看到的入口。

runtime 层

`runtime/dobot_bridge.py` 负责真机执行、预检、dry-run 和复杂长程能力，是插件内部更重的执行层。

适配层

`runtime/robot_factory.py`、`runtime/cellbot_adapter.py` 等文件负责不同机器人 family 的具体适配。

这种结构的好处是边界很清楚：主仓库只关心统一 Driver 接口，插件仓库则负责重依赖、桥接脚本和真机实现。

7. 几种调用方式分别适合什么阶段

方式 适合阶段 优点 注意点
通过 Watchdog + Agent 调用 完整链路验证、演示、系统联调 最接近真实使用场景 问题来源可能横跨 Agent、协议和 runtime 三层
直接写 `ACTION.md` 动作映射测试、Driver 排查 可以绕过 Agent，快速验证动作是否被接住 仍依赖 Watchdog 正常运行
直接调 runtime 插件开发、第三方 SDK 排查、桥接脚本验证 故障范围最小，定位最快 不能代表 Agent 全链路已经成立
这一页的重点不是介绍“插件是什么”，而是帮助你判断在使用、联调和排查阶段，应该采用哪一种插件调用方式。
